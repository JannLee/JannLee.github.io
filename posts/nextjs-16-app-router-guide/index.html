Next.js App Router 완전 정복 — 16 새기능과 마이그레이션 가이드

Next.js App Router 완전 정복 — 16 새기능과 마이그레이션 가이드

Next.js가 App Router를 정식 도입하면서 프런트엔드 개발 패러다임이 크게 바뀌었습니다. React Server Components(RSC), 중첩 레이아웃, 스트리밍, Server Actions, 그리고 Next.js 16에서 도입된 use cache 지시어 — 이 모든 기능이 App Router를 중심으로 통합됩니다.

이 글은 nextjs app router, next.js 16 새기능, react server components 한국어 키워드를 타겟으로, 개념부터 실전 마이그레이션까지 한 번에 정리합니다.

요점 정리

  • App Router는 app/ 디렉터리 기반으로 동작하며, 서버 컴포넌트가 기본값입니다.
  • Pages Router 대비 초기 JS 번들이 평균 40~60% 감소합니다.
  • Server Actions로 별도 API 라우트 없이 폼 처리와 데이터 변경이 가능합니다.
  • Next.js 16의 use cache 지시어가 unstable_cache와 revalidate 설정을 대체합니다.
  • 점진적 마이그레이션 — app/과 pages/를 동시에 사용할 수 있습니다.

1. App Router 핵심 구조

파일 시스템 라우팅

App Router에서는 app/ 디렉터리 내의 파일 구조가 곧 URL 구조입니다. 특수 파일명이 역할을 결정합니다.

app/
├── layout.tsx          # 루트 레이아웃 (필수)
├── page.tsx            # / 라우트
├── loading.tsx         # Suspense 로딩 UI
├── error.tsx           # 에러 바운더리
├── not-found.tsx       # 404 UI
├── blog/
│   ├── layout.tsx      # /blog/* 공유 레이아웃
│   ├── page.tsx        # /blog 라우트
│   └── [slug]/
│       └── page.tsx    # /blog/:slug 동적 라우트
└── api/
    └── hello/
        └── route.ts    # API Route Handler

서버 컴포넌트 vs 클라이언트 컴포넌트

App Router에서 모든 컴포넌트는 기본적으로 서버 컴포넌트입니다. 클라이언트 컴포넌트로 만들려면 파일 상단에 "use client" 지시어를 추가합니다.

보안 팁: HTML 문자열을 직접 렌더링해야 하는 경우 반드시 isomorphic-dompurify 같은 라이브러리로 먼저 sanitize하세요. 신뢰할 수 없는 소스의 HTML을 그대로 주입하면 XSS 공격에 노출됩니다.

핵심 규칙: 서버 컴포넌트는 클라이언트 컴포넌트를 import할 수 있지만, 반대는 불가능합니다. 클라이언트 컴포넌트에서 서버 컴포넌트가 필요하다면 children prop으로 전달하세요.

3. Next.js 16 새기능: use cache 지시어

Next.js 16의 가장 큰 변화 중 하나는 use cache 지시어입니다. 기존의 unstable_cache(), revalidate export, fetch 캐시 옵션을 하나의 일관된 모델로 통합합니다.

활성화 방법

파일/컴포넌트/함수 레벨 캐싱

캐시 무효화

unstable_cache → use cache 마이그레이션

차이점: use cache는 함수 인자와 클로저 변수를 자동으로 캐시 키에 포함시키므로 keyParts 배열이 필요 없습니다.

4. 중첩 레이아웃과 병렬 라우트

중첩 레이아웃

App Router의 강점 중 하나는 레이아웃 중첩입니다. 각 라우트 세그먼트마다 독립적인 레이아웃을 정의합니다.

병렬 라우트 (Parallel Routes)

@ 접두어 폴더로 같은 레이아웃에서 여러 페이지를 동시에 렌더링할 수 있습니다.

app/
└── dashboard/
    ├── layout.tsx
    ├── page.tsx
    ├── @analytics/
    │   └── page.tsx     # 분석 패널 (병렬 렌더링)
    └── @team/
        └── page.tsx     # 팀 패널 (병렬 렌더링)

5. 스트리밍과 Suspense

App Router는 HTTP 스트리밍을 기본 지원합니다. Suspense로 감싸면 나머지 UI를 먼저 전달하고 데이터가 준비되면 순차적으로 채웁니다.

loading.tsx 파일을 만들면 라우트 레벨에서 자동으로 Suspense 바운더리가 생성됩니다.

병렬 데이터 패칭으로 워터폴 제거

독립적인 데이터 패칭은 반드시 병렬로 실행하세요. 순차 await는 불필요한 지연을 만듭니다.

6. Server Actions

Server Actions는 클라이언트에서 서버 함수를 직접 호출하는 방법입니다. 별도의 API 라우트가 필요 없습니다.

Server Actions는 JavaScript가 비활성화된 환경에서도 네이티브 HTML <form> 동작으로 폴백합니다 — Progressive Enhancement 자동 지원입니다.

위 수치는 Vercel 공개 벤치마크 및 실 서비스 마이그레이션 사례 평균입니다. 서비스 특성에 따라 다를 수 있습니다.

RSC의 가장 큰 이점은 컴포넌트 레벨 트리 쉐이킹입니다. 서버에서만 쓰는 라이브러리(예: @prisma/client, fs, 무거운 마크다운 파서)가 클라이언트 번들에 포함되지 않아, 복잡한 앱일수록 번들 절감 효과가 커집니다.

9. 자주 묻는 질문 (FAQ)

Q1. 서버 컴포넌트에서 useState나 useEffect를 쓸 수 없나요?

A. 맞습니다. useState, useEffect, 이벤트 핸들러 등 브라우저 API와 상호작용하는 훅은 클라이언트 컴포넌트에서만 사용 가능합니다. 상태가 필요한 부분만 "use client" 컴포넌트로 분리하고, 나머지는 서버 컴포넌트로 유지하는 것이 베스트 프랙티스입니다.

Q2. getServerSideProps는 App Router에서 어떻게 대체하나요?

A. 서버 컴포넌트 자체가 서버에서 실행되므로 별도의 함수가 필요 없습니다. 컴포넌트를 async로 선언하고 내부에서 직접 await로 데이터를 패칭하면 됩니다. 매 요청마다 실행되는 동적 렌더링이 필요하다면 fetch(url, { cache: "no-store" })를 사용하거나 cookies() / headers() 같은 동적 API를 호출하면 Next.js가 자동으로 동적 렌더링으로 전환합니다.

Q3. App Router에서 getStaticProps의 revalidate는 어떻게 쓰나요?

A. Next.js 16부터는 use cache 지시어로 처리합니다.

이전 버전에서는 Route Segment Config로 설정했습니다: export const revalidate = 3600

Q4. Pages Router와 App Router를 동시에 사용할 수 있나요?

A. 가능합니다. 같은 URL 경로를 두 디렉터리에 중복 정의하는 것만 피하면 됩니다. pages/about.tsx가 있으면 app/about/page.tsx를 동시에 만들 수 없습니다. 점진적으로 라우트를 이전하는 방식을 Next.js 공식 문서에서도 권장합니다.

Q5. next/image와 next/link 사용법이 바뀌었나요?

A. next/link는 App Router에서 더 이상 <a> 태그를 자식으로 요구하지 않습니다 (Next.js 13 이후). next/image는 width/height를 여전히 필요로 하거나, fill prop으로 부모 컨테이너를 채울 수 있습니다.

Q6. 전역 상태 관리(Redux, Zustand)는 어떻게 쓰나요?

A. Provider를 "use client" 컴포넌트로 래핑하고, app/layout.tsx에서 children을 그 Provider로 감싸는 패턴이 일반적입니다.

Q7. 환경 변수 접근 방식이 달라졌나요?

A. 기본 원칙은 동일합니다. NEXT_PUBLIC_ 접두어가 없는 환경 변수는 서버에서만 접근 가능합니다. App Router에서는 서버 컴포넌트가 기본이므로, 실수로 비밀 변수가 클라이언트에 노출될 위험이 Pages Router보다 낮습니다. "use client" 컴포넌트에서 NEXT_PUBLIC_ 없는 환경 변수를 참조하면 빌드 타임에 에러가 발생합니다.

Q8. middleware.ts는 App Router에서도 동작하나요?

A. 동작합니다. middleware.ts는 app/과 pages/ 모두와 독립적으로 프로젝트 루트에 위치하며, 라우트 처리 전에 실행됩니다. 인증 체크, 리디렉션, A/B 테스트, 지역화 등에 활용합니다.

정리

Next.js App Router는 단순한 라우팅 개선이 아닙니다. React Server Components를 중심으로 한 새로운 렌더링 모델의 도입이며, 이는 번들 크기, 성능, 데이터 패칭 방식 전반에 걸친 패러다임 전환입니다. Next.js 16의 use cache 지시어는 그 완성도를 한 단계 더 높입니다.

마이그레이션은 점진적으로 진행할 수 있습니다:

  1. app/layout.tsx와 루트 레이아웃 먼저 설정
  2. 신규 페이지는 app/에 작성
  3. 트래픽이 낮은 기존 페이지부터 순차적으로 이전
  4. 데이터 패칭 패턴을 컴포넌트 내 async/await로 통일
  5. unstable_cache → use cache 지시어로 마이그레이션
  6. 클라이언트 상태가 필요한 부분만 "use client" 분리

App Router의 학습 곡선은 분명 존재하지만, 익숙해지고 나면 개발 생산성과 최종 사용자 경험 모두 크게 향상됩니다. 특히 서버 컴포넌트 덕분에 "API 라우트 없이 DB에서 바로 데이터를 가져오는" 직관적인 풀스택 개발이 가능해집니다.
