GitHub Pages + Next.js 블로그 완전 배포 가이드 (CI/CD 자동화)

Next.js output: 'export'를 사용해 GitHub Pages에 블로그를 배포하는 전체 과정을 정리합니다. GitHub Actions 자동화부터 트러블슈팅, 배포 시간 최적화까지 단계별로 설명합니다.

next.config.ts 핵심 설정

  • output: "export" — 빌드 결과물을 정적 HTML/CSS/JS로 내보냅니다
  • trailingSlash: true — GitHub Pages의 경로 규칙과 맞춥니다
  • images.unoptimized: true — Next.js Image Optimization은 서버가 필요하므로 비활성화

GitHub Actions 배포 설정

.github/workflows/deploy.yml을 생성해 main 브랜치 푸시 시 자동 배포되도록 합니다.

빌드 캐싱 전략

actions/setup-node@v4의 cache: pnpm 옵션은 pnpm store 경로를 자동으로 캐싱합니다. 이를 통해 의존성 설치 시간을 대폭 줄일 수 있습니다.

  • 캐시 없을 때 (첫 실행): pnpm install 약 2분 30초
  • 캐시 적용 후: pnpm install 약 15~30초
  • 전체 빌드 시간: 캐시 미적용 시 약 4분 → 캐시 적용 시 약 1분 30초

환경 변수 관리

GitHub Actions에서 환경 변수를 다루는 방법은 두 가지입니다.

종류설정 위치용도
secretsRepository Settings > SecretsAPI 키, 토큰 등 민감 정보
env워크플로우 파일 env: 블록공개 가능한 설정값

Next.js에서 빌드 시점에 주입되는 변수는 반드시 NEXT_PUBLIC_ 접두사를 붙여야 클라이언트에서 접근 가능합니다. 서버 전용 변수는 output: 'export' 환경에서 사용할 수 없습니다.

브랜치 전략

  • main 브랜치 보호: Settings > Branches에서 보호 규칙 추가 (PR 필수, 직접 push 차단)
  • dev 브랜치 preview: Vercel 또는 Cloudflare Pages를 연동해 PR마다 preview URL 생성

배포 시간 벤치마크

실제 운영 중인 Next.js 블로그 기준 측정값입니다.

단계시간
코드 체크아웃~5초
pnpm install (캐시 미적용)~2분 30초
pnpm install (캐시 적용)~20초
Next.js 빌드 (pnpm build)~45초
out/ 폴더 업로드~30초
GitHub Pages 배포~15초
전체 (캐시 적용 기준)약 2~3분

캐시를 처음 적용하면 첫 실행은 약 4분이 소요되고, 이후 실행부터는 지속적으로 2~3분 이내로 단축됩니다.

자주 겪는 오류 5가지

1. _next/ 경로 404 오류

오류 메시지: 404 Not Found — /_next/static/chunks/...

원인: GitHub Pages는 기본적으로 Jekyll을 통해 정적 파일을 서빙하는데, Jekyll은 _로 시작하는 폴더를 빌드 대상에서 제외합니다.

해결책: public/ 폴더에 빈 .nojekyll 파일을 추가합니다.

2. 이미지 로드 실패

오류 메시지: 이미지가 빈 박스로 표시되거나 콘솔에 404 오류

원인: next.config.ts에 images.unoptimized: true가 없으면 정적 내보내기 시 이미지 최적화 API 호출이 발생합니다.

원인: 레포지토리 이름이 username.github.io가 아닌 경우, GitHub Pages URL은 https://username.github.io/repo-name/이지만 basePath 설정이 없으면 링크가 루트(/)를 기준으로 생성됩니다.

고급 설정

커스텀 도메인 설정

  1. public/ 폴더에 CNAME 파일 생성:
yourdomain.com

  1. DNS 설정 (도메인 등록업체 관리 패널):

    • A 레코드: 185.199.108.153, 185.199.109.153, 185.199.110.153, 185.199.111.153
    • CNAME: www → username.github.io

  2. Repository Settings > Pages > Custom domain에 도메인 입력 후 HTTPS 강제 활성화

PR Preview 배포

GitHub Pages는 단일 URL만 제공하므로 PR preview를 원한다면 추가 서비스가 필요합니다.

  • Vercel: 레포 연동 후 모든 PR에 자동 preview URL 생성 (무료 플랜 가능)
  • Cloudflare Pages: 비슷한 기능 제공, CDN 성능 우수

요약

  • Next.js 블로그를 GitHub Pages에 배포하려면 output: 'export', trailingSlash: true, images.unoptimized: true 세 가지 설정이 필수입니다
  • .nojekyll 파일 추가로 _next/ 정적 자산 경로 404를 방지합니다
  • GitHub Actions + pnpm 캐시 조합으로 배포 시간을 2~3분 이내로 유지할 수 있습니다
  • main 브랜치 보호와 permissions 명시로 안전한 자동 배포 파이프라인을 구성합니다
  • 트러블슈팅 5가지를 숙지하면 대부분의 배포 문제를 빠르게 해결할 수 있습니다

FAQ

Q. GitHub Pages는 무료인가요? 공개 레포지토리는 무료입니다. 비공개 레포지토리는 GitHub Pro 이상 플랜이 필요합니다. 트래픽 제한은 월 100GB 대역폭, 소프트 한도 1GB 스토리지입니다.

Q. 커스텀 도메인을 붙이고 싶어요. public/CNAME 파일에 도메인을 적고, DNS에 GitHub Pages IP를 A 레코드로 등록합니다. HTTPS는 Let's Encrypt를 통해 자동 적용됩니다 (DNS 전파 후 약 10~15분 소요).

Q. Next.js 이외에 다른 프레임워크도 배포할 수 있나요? 네. Astro, Hugo, Jekyll, Gatsby 등 정적 파일을 출력하는 모든 프레임워크가 가능합니다. GitHub Pages의 핵심은 정적 파일 호스팅이므로 프레임워크 종류와 무관합니다.

Q. 배포 시간이 너무 오래 걸려요. pnpm 캐시(cache: pnpm)와 Next.js 빌드 캐시(.next/cache)를 모두 활성화하세요. 두 캐시를 함께 적용하면 반복 배포 시간을 약 4분에서 2분 이내로 줄일 수 있습니다.

Q. 빌드가 실패했을 때 어디서 확인하나요? GitHub 레포지토리 > Actions 탭에서 실행 로그를 확인할 수 있습니다. 각 스텝별로 펼쳐서 보거나, 우측 상단 "Download log archive"로 전체 로그를 내려받을 수 있습니다.

Q. HTTPS는 자동으로 적용되나요? github.io 기본 도메인은 HTTPS가 자동 적용됩니다. 커스텀 도메인도 Settings > Pages에서 "Enforce HTTPS"를 활성화하면 Let's Encrypt 인증서가 자동 발급됩니다.

Q. 비공개 레포에서도 GitHub Pages를 쓸 수 있나요? GitHub Pro, Team, Enterprise 플랜에서 비공개 레포의 GitHub Pages를 사용할 수 있습니다. 단, 배포된 사이트 자체는 공개됩니다 — 소스 코드가 비공개일 뿐, 빌드된 HTML은 누구나 접근 가능합니다.

결론

GitHub Pages + GitHub Actions 조합은 블로그 배포를 위한 가장 간단하고 비용 효율적인 방법 중 하나입니다. next.config.ts 설정 3줄과 워크플로우 파일 하나로 코드 push와 동시에 자동 배포가 이루어집니다. pnpm 캐시를 활용하면 배포 2~3분 이내를 안정적으로 달성할 수 있습니다.
