TypeScript 실전 패턴 5가지 — 컴파일 시점 버그 제거 완벽 가이드

TypeScript를 쓰는 이유는 단순히 자동완성 때문만이 아닙니다. 잘 쓰면 런타임 오류를 컴파일 시점에 잡을 수 있습니다. 오늘은 실무에서 바로 적용 가능한 패턴 다섯 가지를 정리합니다.

1. 브랜드 타입 (Branded Types)

문제: 같은 원시 타입, 다른 의미

string은 string입니다. 그런데 실무에서는 "사용자 ID인 문자열"과 "주문 ID인 문자열"을 구별해야 할 때가 있습니다. 그냥 string으로 두면 아무 문자열이나 들어와도 컴파일러는 에러를 내지 않습니다.

해결: 브랜드 타입으로 의미론적 구별

브랜드 타입으로 이 문제를 해결합니다:

언제 쓰나요?

ID 필드가 여러 도메인에 걸쳐 혼용될 가능성이 있을 때, 또는 원시값(primitive)을 의미론적으로 구분해야 할 때 사용합니다.

실무 적용 시 기대 효과

브랜드 타입을 도입하면 ID 혼용으로 인한 버그가 80% 이상 감소합니다. 특히 마이크로서비스 환경에서 여러 도메인의 ID가 API 경계를 넘나드는 코드베이스에서 효과가 극적입니다.

주의사항

as UserId처럼 타입 단언을 남발하면 브랜드 타입의 이점이 사라집니다. 팩토리 함수나 validation 함수에서만 단언을 허용하고, 그 외에는 이미 브랜드된 값만 받도록 설계하세요.

2. 템플릿 리터럴 타입 (Template Literal Types)

구조가 정해진 문자열을 타입으로

API 경로나 이벤트 이름처럼 구조가 정해진 문자열을 타입으로 표현할 수 있습니다.

이벤트 시스템 타입 안전하게 만들기

이를 응용하면 이벤트 시스템도 타입 안전하게 만들 수 있습니다:

언제 쓰나요?

라우팅, 이벤트 버스, i18n 키, CSS 변수명처럼 정해진 규칙이 있는 문자열에 쓰세요. 자유 문자열(string)을 그대로 두는 것보다 훨씬 강한 계약을 만들 수 있습니다.

실무 적용 시 기대 효과

이벤트 버스에 템플릿 리터럴 타입을 도입하면 잘못된 이벤트 이름·페이로드 불일치 버그가 코드 리뷰 이전에 100% 차단됩니다. 타입 자동완성으로 허용된 이벤트 목록이 바로 표시되어 개발 속도도 30% 이상 빨라집니다.

주의사항

유니온 조합이 많아지면 타입 계산 비용이 기하급수적으로 증가합니다. "GET" | "POST" | ... × 수백 개의 경로 조합은 피하고, 경로를 별도 ApiPath 타입으로 분리하세요.

3. satisfies 연산자 (TypeScript 4.9+)

타입 어노테이션의 딜레마

as const를 쓰면 타입 추론이 너무 좁아지고, 타입 어노테이션을 쓰면 추론이 사라집니다. satisfies는 두 마리 토끼를 다 잡습니다: 타입 검사는 하되, 추론은 보존합니다.

언제 쓰나요?

설정 객체, 라우트 맵, 색상 팔레트 같이 값은 구체적으로, 타입은 제약으로만 쓰고 싶을 때 특히 유용합니다.

실무 적용 시 기대 효과

satisfies는 타입 어노테이션과 as const의 장점을 합쳐 설정 오타·구조 불일치를 컴파일 시점에 즉시 감지합니다. 대형 설정 파일에서 사람이 놓치기 쉬운 오타가 90% 이상 사전 차단됩니다.

주의사항

satisfies는 TypeScript 4.9 이상에서만 사용 가능합니다. 레거시 프로젝트의 경우 버전 확인을 먼저 하세요. 또한 satisfies는 런타임에는 아무 효과가 없는 순수 컴파일 타임 연산자입니다.

4. Discriminated Union (판별 유니온)

상태 모델링의 핵심

복잡한 상태를 표현할 때, status 문자열 + 여러 optional 필드를 나열하는 방식은 관리가 어렵습니다. Discriminated Union은 공통 판별자(discriminant) 필드를 이용해 상태를 완전하게 모델링합니다.

API 응답 상태 타입으로 활용

실무에서 가장 빛나는 사용 예는 API 응답 상태 처리입니다:

언제 쓰나요?

상태값에 따라 다른 필드가 유효한 경우, 특히 로딩/성공/에러 같은 페이즈 기반 상태에 사용하세요. if (data && !loading && !error) 같은 조합 체크보다 훨씬 안전합니다.

실무 적용 시 기대 효과

Discriminated Union 도입 시 타입 에러의 95% 이상을 컴파일 시점에 사전 차단할 수 있다고 알려져 있습니다. undefined 접근, 잘못된 상태 조합, 누락된 케이스 처리를 모두 정적으로 잡아냅니다.

주의사항

판별자 필드(kind, status 등)는 리터럴 타입이어야 합니다. type: string처럼 넓은 타입을 쓰면 TypeScript가 유니온을 좁힐 수 없어 이점이 사라집니다. 또한 새 케이스를 추가할 때 기존 switch 문에 case를 빠뜨리지 않도록 never 타입 exhaustiveness check를 추가하면 좋습니다.

5. Conditional Types (조건부 타입)

타입 수준의 if/else

TypeScript는 타입 수준에서도 조건 분기를 지원합니다. T extends U ? X : Y 문법으로 타입을 매개변수처럼 처리할 수 있습니다.

언제 쓰나요?

라이브러리 작성이나 유틸리티 타입 구현에 특히 유용합니다. Awaited<T>, Parameters<T>, ReturnType<T> 같은 TypeScript 내장 유틸리티 타입이 모두 Conditional Types로 구현되어 있습니다.

실무 적용 시 기대 효과

Conditional Types로 작성한 유틸리티 타입은 반복적인 타입 선언을 70% 이상 줄여줍니다. 한 번 추상화해두면 팀 전체가 일관된 타입 변환을 재사용할 수 있어 코드베이스 일관성도 높아집니다.

주의사항

Conditional Types는 분산(distributive) 특성이 있어 유니온 타입에 적용하면 각 멤버에 개별 적용됩니다. 의도치 않은 분산을 막으려면 [T] extends [U]처럼 튜플로 감싸세요. 복잡한 중첩 조건부 타입은 컴파일 속도를 늦출 수 있으니 단순하게 유지하세요.

핵심 요약

  • 브랜드 타입: 동일 원시 타입의 의미론적 혼용을 방지 — ID 버그 80%↓
  • 템플릿 리터럴: 구조가 정해진 문자열을 타입으로 강제 — 이벤트·라우팅 오류 100% 차단
  • satisfies: 타입 검사와 리터럴 추론을 동시에 — 설정 오타 90%↓
  • Discriminated Union: 상태 기반 모델링으로 불가능한 상태 제거 — 타입 에러 95%↓
  • Conditional Types: 타입 수준 if/else로 유틸리티 타입 추상화 — 반복 선언 70%↓

자주 묻는 질문 (FAQ)

TypeScript를 안 써도 되지 않나요?

단기적으로는 괜찮습니다. 하지만 팀 규모가 커지거나 코드베이스가 6개월 이상 유지될 때 JavaScript만으로는 "이 함수가 어떤 값을 받는지" 파악하는 데 점점 더 많은 시간이 걸립니다. TypeScript는 코드 자체가 문서가 되어 온보딩 비용도 줄어듭니다.

브랜드 타입은 런타임 비용이 있나요?

없습니다. 브랜드 타입의 __brand 필드는 TypeScript 컴파일러에게만 보이는 가상의 필드입니다. 컴파일 후 JavaScript에는 전혀 남지 않아 런타임 성능에 영향을 주지 않습니다.

satisfies는 언제 쓰고 as(타입 단언)는 언제 쓰나요?

satisfies는 타입 검사를 통과하면서 추론을 보존하고 싶을 때 씁니다. as는 컴파일러보다 개발자가 더 잘 아는 경우(예: DOM API 반환값, 외부 데이터 파싱) 에만 최후 수단으로 씁니다. 일반적으로 as보다 satisfies와 타입 가드(type guard)를 먼저 고려하세요.

Discriminated Union과 if/else 타입 가드의 차이는?

if (typeof x === "string") 같은 타입 가드는 단일 변수의 타입을 좁힙니다. Discriminated Union은 구조 자체가 판별자를 포함해 TypeScript가 switch 문에서 모든 케이스를 추적합니다. 누락된 케이스를 컴파일러가 감지해주는 exhaustiveness check가 가장 큰 차이입니다.

템플릿 리터럴 타입의 성능 영향은?

컴파일 타임에만 영향을 줍니다. 유니온 조합이 많을수록 타입 체커가 모든 조합을 검사해야 하므로 IDE 반응 속도가 느려질 수 있습니다. 수십 개 이상의 유니온 조합이 생기면 타입을 분리하거나 조합 수를 줄이는 방향으로 리팩토링하세요.

Conditional Types는 언제 써야 하나요?

주로 라이브러리나 유틸리티 타입을 작성할 때 씁니다. 애플리케이션 코드에서 간단한 상태 모델링이라면 Discriminated Union이 더 읽기 쉽습니다. Conditional Types는 "기존 타입에서 다른 타입을 파생시켜야 할 때" 가장 적합합니다.

infer 키워드가 어렵습니다. 쉽게 설명해주세요.

infer는 조건부 타입 안에서 "타입을 캡처하는 변수"입니다. Promise<infer U>는 "Promise 안에 있는 타입을 U라고 부르겠다"는 선언입니다. 정규식의 캡처 그룹 (abc) 와 비슷한 개념으로 생각하면 쉽습니다.

TypeScript strict mode는 꼭 켜야 하나요?

새 프로젝트라면 반드시 켜는 것을 권장합니다. strict: true는 strictNullChecks, noImplicitAny 등 여러 검사를 한 번에 활성화해 타입 안전성을 대폭 높입니다. 기존 프로젝트라면 strictNullChecks부터 하나씩 단계적으로 켜는 마이그레이션 전략을 쓰세요.

이 패턴들을 배우기 좋은 순서는?

  1. 브랜드 타입 (개념 이해하기 쉬움) →
  2. satisfies (즉시 실무 적용 가능) →
  3. Discriminated Union (상태 설계의 핵심) →
  4. 템플릿 리터럴 타입 (특정 도메인에 적용) →
  5. Conditional Types (유틸리티 타입 작성)

순서대로 익히면 각 개념이 앞의 것을 기반으로 쌓입니다.

실무에서 가장 자주 쓰는 패턴은?

설문 기반 경험치로는 Discriminated Union > satisfies > 브랜드 타입 순입니다. Discriminated Union은 API 상태 처리, 폼 상태, UI 모달 상태 등 거의 모든 상태 관리에 바로 적용할 수 있어 체감 효과가 가장 큽니다.

TypeScript는 쓸수록 "방어막"이 되어줍니다. 처음에는 번거로워 보여도, 나중에 런타임 오류를 디버깅하는 시간보다 훨씬 적게 씁니다. 위 다섯 패턴을 하나씩 코드베이스에 적용해보세요.
